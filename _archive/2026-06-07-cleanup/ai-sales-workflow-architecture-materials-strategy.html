AI-powered sales workflow — system architecture

How the CRM, Outlook, meeting-transcript tool and dashboard talk to each other and to Claude — four integrated workflows, one orchestration layer, human-in-the-loop on every write for the first month.

Author: Rabii + Claude · 23 May 2026 · Status: Proposal for review · Confidential

Status

Proposal for review. Two key unknowns marked <NEEDS CONFIRMATION> — answer those and the build plan can be locked. Companion to the Automations plan: that document covered file-based automations inside the StableSensei tree; this one specifies how the CRM, Outlook, meeting-transcript tool and dashboard talk to each other and to Claude. The two are designed to coexist — file-based tooling stays for narrative work (drafts, account plans, post-mortems); CRM is system-of-record for pipeline state.

1. Summary

Four integrated workflows, one orchestration layer, human-in-the-loop on every write for the first month.

The proposal is to use Microsoft Graph for Outlook, the CRM's own REST/GraphQL API for lead and activity sync, and a transcript-tool webhook (<NEEDS CONFIRMATION: Fireflies / Gong / Granola / Teams Premium>) for meeting capture. The orchestrator that glues them together is the smallest piece that works — for the first phase that means n8n (self-hostable, open source, predictable) with Claude API doing the language-heavy lifting. The dashboard is a Cowork live artifact that reads from the CRM, so it never holds state itself and stays consistent with what the CRM says.

The non-obvious design choices are: never auto-delete email, never let Claude write to the CRM without a 48-hour human-review window for the first month, and treat the CRM's status fields as the only authoritative pipeline view. Files under Sales/[Product]/Cases/ remain the place where prose lives; CRM is where status and ownership live. Mixing the two has caused enough confusion for other teams that it's worth being explicit up front.

Three rules govern the diagram:

The orchestrator never holds canonical data. It moves messages, applies rules, calls Claude, and writes results to the CRM or the file tree. If n8n crashes, nothing is lost — sources of truth are upstream (Outlook, transcript tool) and downstream (CRM, files).

Claude API is the language layer. It classifies, synthesises, and drafts. It does not decide. Decisions (delete, send, change stage) happen either in n8n via deterministic rules, or by a human action on a Claude-prepared draft.

The dashboard reads, it does not write. A live Cowork artifact loads the CRM's current state on every open. That keeps it cheap to build, impossible to drift, and easy to throw away if the CRM's native dashboard improves.

Priority is a rule, not a guess

P1 = (Stage in [commercial, legal, pilot]) AND (Days Since Touch >= 3)
     OR (Next Action Deadline <= today)
     OR (manually flagged by Rabii)

P2 = (Stage in [discovery, scoping]) AND (Days Since Touch >= 5)
     OR (Next Action Deadline within 7 days)
     OR (high-value account: see Account.Tier = A)

P3 = everything else with an open Next Action

P4 = parked / no next action

Priority is recomputed nightly by the orchestrator. Rabii can override manually; the override sticks for 7 days then re-evaluates.

Activity records

Every CRM-note becomes an Activity attached to the Lead/Opportunity. Activity types: email-in, email-out, meeting, crm-note, ai-summary. Each activity carries the source (e.g. meeting:2026-05-22:fireflies-recording-id) so we can always trace back.

5. The four workflows

5.1 CRM integration

Trigger sources: new email from an unknown sender at a domain we recognise; transcript-tool webhook; manual command from Rabii.

Flow for a new lead from email:

  1. Inbound email lands in Outlook. Microsoft Graph subscription fires a webhook to n8n.
  2. n8n extracts sender, domain, subject, body. Calls Claude (Sonnet) to classify intent: is this commercial outreach to Aryze, a customer service question, a marketing/newsletter mail, or noise?
  3. If commercial: n8n queries CRM for a Contact matching the sender's email. If none exists, query for an Account matching the sender's domain.
  4. If neither exists, n8n drafts a new Lead record (Claude produces the One-line Summary, picks the Aryze Product based on email content, picks Trigger Event from the catalogue). Confidence score is attached.
  5. Draft is held in the review queue. Rabii sees it in the dashboard the next morning (or sooner via push). One click to approve → it's written to CRM.
  6. Existing Lead/Contact: n8n appends an Activity record (email-in), and if the email contains a concrete next step (Claude extracts), it proposes an update to Next Action. Held for review if confidence <70.

Flow for transcript-driven updates: see §5.3.

Idempotency: every external event gets a deterministic ID (outlook-message-id, fireflies-recording-id, etc.) stored on the Activity. Reprocessing the same event is a no-op.

5.2 Outlook inbox management

This is where the biggest risks live. Email is irreversible; a wrongly-deleted message is gone and not recoverable in a meaningful way after 30 days. The design below is deliberately cautious.

What runs automatically without human review:

Categorisation only. n8n applies Outlook Categories to incoming mail based on Claude's classification:

  • Aryze: Customer — known customer/prospect domain
  • Aryze: Internal — @aryze.io sender
  • Aryze: Vendor — known vendor domain (Volt, Mastercard, etc.)
  • Aryze: Newsletter — Claude classified as newsletter/promotional
  • Aryze: Likely-Spam — Claude classified as spam-like, low confidence
  • Aryze: Needs Reply — Claude detected a question or request directed at Rabii

Outlook surfaces these as coloured tags. A view filter sorted by category puts "Needs Reply" at the top.

What runs automatically with a human-built allow/block list:

Auto-move to subfolders only for senders Rabii has explicitly approved. The dashboard surfaces "candidates to add to your block list" (senders flagged as Newsletter 3+ times) — Rabii approves; n8n then moves future mail from that sender to \Newsletters automatically. No deletes.

What never runs automatically:

Auto-delete. Auto-reply. Auto-forward. Auto-unsubscribe (clicking unsubscribe links is a known attack vector — bad senders use unsubscribe clicks as confirmation that the mailbox is live).

Senders to "block":

Outlook's native block-sender rule moves mail to Junk. n8n proposes additions to the block list weekly; Rabii approves a batch. No silent additions.

Flag for follow-up:

Claude flags emails that need a reply. n8n applies Outlook's red flag and the Aryze: Needs Reply category. Anything not flagged stays unflagged — better to have Rabii glance over an unflagged email than to over-flag and lose signal.

Confidence thresholds: classify-only ≥60. Move to subfolder ≥85. Delete: never auto.

5.3 Meeting transcript workflow

Meeting selection — opt-in by design. Not every meeting is a sales meeting. Internal team syncs, HR conversations, partner check-ins, vendor admin calls, personal events — none of these should be recorded or processed by the AI listener. Over-capture creates noise in the CRM-note pipeline, costs money per minute transcribed, and makes the system feel like surveillance to people who didn't sign up to be transcribed. For a Compliance-branded company the right default is the cautious one.

The principle: the AI listener never auto-joins. Rabii decides per meeting. How that decision is registered depends on the tool:

ToolOpt-in mechanismVisibility to other attendeesRecommended setting
GranolaRuns locally on Rabii's Mac; Rabii starts it manuallyNone — no bot, no avatar, nothing visibleAlready opt-in by construction. Nothing to disable.
FirefliesBot (e.g. fred@fireflies.ai) added as calendar attendeeBot appears in attendee list and joins as a visible participantDisable "auto-join all meetings on my calendar" in user settings. Bot only joins when explicitly invited.
GongAdmin-level rules, typically auto-record external meetingsNo bot, recording is server-side on Zoom/Teams/MeetDisable "auto-record all external meetings". Record only when Rabii explicitly tags the meeting or invites the Gong bot.
Teams PremiumMeeting organiser clicks record per meetingNative Teams record indicatorOpt-in by default. No configuration change needed.

Whatever tool we land on, the rule is the same: no recording without an explicit per-meeting signal from Rabii.

Belt and braces — second filter at the orchestrator. Even when a recording arrives at n8n, the workflow validates before calling Claude. Three checks:

  1. Was there at least one external attendee (non-@aryze.io email) on the calendar invite?
  2. Was the meeting at least 5 minutes long?
  3. Does the meeting title, description, or attendee list match an Aryze commercial context (known customer/prospect name, an existing case folder under Sales/[Product]/Cases/, or an explicit [Sales] / [Aryze] tag in the title)?

If two of the three are no, the recording is logged but not processed. It appears in a "Skipped" list in the dashboard. Rabii can promote it to processing with one click if the note is wanted anyway — e.g. an internal strategy call where the action items should be extracted but not posted to a customer CRM record.

This handles the two failure modes that opt-in alone doesn't: the bot was invited to a meeting that turned out not to be commercial, and a recording exists but the conversation wasn't sales-relevant.

A note on tool choice in light of opt-in. Granola has the strongest opt-in posture (local, no visible bot, customer never sees an AI participant) which matches Aryze's brand voice — discreet, operational, compliance-first. Fireflies has better out-of-the-box CRM integrations but the visible bot can change the dynamic of a meeting, particularly with regulated counter-parties (banks, FCA-supervised customers) who may object to a third-party recorder on the call. Gong is over-configured for Aryze's stage. Teams Premium is the safest from a vendor-concentration angle but its CRM-side automation is the weakest of the four. Recommendation lean: Granola or Teams Premium for opt-in cleanliness, with Granola preferred if its API maturity is enough for the n8n integration. Confirm in the decision checklist below.

Trigger: transcript tool posts a webhook (or, for Teams Premium, a Graph subscription on callRecords fires) — but only for meetings that passed the opt-in step at the source.

Flow:

  1. Webhook hits n8n with recording ID + transcript URL.
  2. n8n fetches the transcript and meeting metadata (attendees, duration, calendar invite).
  3. n8n calls Claude (Opus) with a structured prompt anchored on Brain/Templates/CRM-Note.md and the relevant Sales persona. Output is a strict JSON object with these fields: customer; aryze_product (Pay by Bank | Stablecoin Factory | Digital Cash | unclear); summary_1_line (≤200 chars); latest (≤3 lines); open_questions; next_action; next_action_deadline; customer_numbers (any volumes, average order values (AOVs), or specifics they cited — use their numbers, don't approximate, per the CRM-Note rule); decision_makers_mentioned (name, title, role on buyer committee); timeline; potential_value (qualitative, no fabricated revenue figures); risks_or_objections; confidence (0–100).
  4. n8n then runs two writes in parallel: a file tree write — save as Sales/[Product]/Cases/[Customer]/CRM-Notes/[YYYY-MM-DD].md formatted per the template (if [Product] or [Customer] is ambiguous, save to a _inbox/ folder and flag for Rabii) — and a CRM write — create or update the Lead/Opportunity, append an Activity (meeting), update Last Touch Date, propose Next Action and Next Action Deadline.
  5. CRM and file writes are held in the review queue if confidence < 80. Held for 48 hours during the first month; after the first month, auto-apply at ≥80 and queue only the <80 ones.
  6. On approval, the dashboard reflects the new state on next load.

Brand voice guard: the synthesis prompt includes the forbidden-phrases list from the master CLAUDE.md ("revolutionising", "seamless", "future of money", etc.) and the voice dialer (mature / sophisticated / economical / literal / contemporary). The CRM-note draft is screened against the list before being queued. The screen is a simple regex over the forbidden list — not Claude self-checking.

Capability discipline guard: if the transcript surfaces a capability claim that hasn't been verified (e.g. "they want AIS", "they want global coverage", "they want real-time payouts"), the AI flags it as an Open Question rather than committing it to the CRM as a confirmed need. The Product-Description files (Documents/Analyser/[Product]/Product-Description.md) are loaded into the synthesis prompt as ground truth on what Aryze actually does today.

5.4 Follow-up and prioritisation

The morning brief is the user-facing surface for this workflow.

Daily 07:00 CEST (weekdays):

  1. n8n queries the CRM for all P1 and P2 leads with Next Action Deadline today or overdue.
  2. Queries Microsoft Graph for the day's calendar.
  3. Queries Outlook for emails in Aryze: Needs Reply from the last 24 hours.
  4. Queries the file tree for stale cases (Days Since Touch ≥ 14).
  5. Claude (Sonnet) assembles a one-page brief in Rabii's voice: today's calls (with prep links), emails needing reply (with one-line context per email), top three deals at risk this week, top three actions to take today.
  6. Brief is saved to Documents/Reports/Daily-Briefs/[date].md and (optional) sent as an Outlook calendar attachment at 07:00.

Sunday 17:30 CEST: the weekly-update auto-draft already specified in the previous plan. Pulls from the same CRM Activity stream now that we have it.

Dashboard "needs action today" view: a live Cowork artifact. Filters CRM for Priority IN (P1, P2) AND Next Action Deadline <= today + 2. Each row links to the case folder and the CRM record.

6. Phased implementation plan

The principle is to make every phase land working software with no regressions before adding the next surface area.

Phase 0 — Discovery and access (week 1). Confirm the CRM and transcript tool. Register an Azure AD application for Microsoft Graph (delegated permissions: Mail.Read, Mail.ReadWrite, MailboxSettings.ReadWrite, Calendars.Read, User.Read). Obtain CRM API credentials and confirm webhook capability. Stand up an n8n instance (small Hetzner VM, EU region, ~EUR 10/month). Stand up a small Postgres for the review queue.

Phase 1 — Read-only path (week 2). n8n subscribes to Outlook inbox. Claude classifies every inbound email. n8n applies Categories only — no moves, no deletes, no flags. A daily summary email lands in Rabii's inbox at 18:00 showing what was classified and how. Goal: validate the classifier against a real week of mail. Adjust prompts until accuracy on a held-out sample ≥ 90% on commercial vs. newsletter, ≥ 80% on "needs reply".

Phase 2 — Transcript path read-only (week 3). Transcript webhook activates. n8n calls Claude, produces draft CRM-notes, writes to file tree only (no CRM write yet). Rabii reviews each draft against what would have been written manually. Adjust the synthesis prompt until drafts need <20% editing for a representative week.

Phase 3 — Selective writes (week 4). Inbox: turn on the red flag for "Needs Reply" category. CRM: enable Activity-record writes (append-only, never overwrite). Transcript: write to CRM as a meeting Activity with status "draft, pending review". Rabii approves each in the dashboard. Still no auto-stage changes, no auto-Next-Action commits.

Phase 4 — Auto-fields with thresholds (week 5). Auto-update Last Touch Date and Days Since Touch. Auto-set Next Action and Next Action Deadline for confidence ≥80 transcripts; queue the rest. Priority computation goes live.

Phase 5 — Dashboard (week 6). Live Cowork artifact reading the CRM. Filters for P1/P2 today, stale cases, "needs reply" inbox. Replaces manual scanning of the file tree.

Phase 6 — Inbox actions (week 7). Auto-move to \Newsletters for senders on the Rabii-approved subfolder list. Block-sender list reviewed weekly. Still no deletes.

Phase 7 — Morning and weekly briefs (week 8). Daily 07:00 brief turns on after one week of dry-run output. Sunday 17:30 weekly auto-draft turns on.

Phase 8 — Iterate and harden (week 9+). Tune confidence thresholds, add the second priority of fields (VPC Pain, Buyer Committee, Trigger Event auto-population), expand to outbound email tracking, consider Outreach.io / Apollo as outbound add-ons if needed.

The schedule assumes ~10 hours/week of build effort. It can compress if Phase 0 access is granted quickly.

7. Risks, permissions, security

The most important risk is irreversible automated actions on email. Auto-delete is excluded entirely from this design. Auto-reply is excluded. Auto-forward is excluded. Block-sender additions require Rabii's approval each week. If those rules feel restrictive in six months, they can be relaxed deliberately — far easier than recovering from a wrongly-deleted contract.

The second-most important risk is the CRM becoming wrong without Rabii noticing. Mitigations are the AI Confidence field (visible in the dashboard, low scores get flagged), the 48-hour human-review window for the first month, and append-only Activity records (so even a wrong update never overwrites history — the prior state is recoverable from the Activity log).

Permissions, principle of least privilege. Microsoft Graph delegated permissions: read mail, read/write categories and flags, read calendar, read user profile. Not Mail.Send until Phase 8 (and only if we decide to enable AI-drafted send-from-Rabii). Not Mail.ReadWrite.Shared. CRM API: a dedicated service-account user with role limited to Lead/Opportunity/Activity read+write — not admin. Azure AD app registration uses certificate-based auth, not client secret. All credentials in n8n's encrypted credential store; backups of n8n's database are encrypted at rest.

GDPR and data residency. Aryze is EU-based; the CRM presumably holds EU personal data. Three actions: (1) execute Anthropic's DPA so AI processing is covered (already standard for Claude API customers; double-check it's signed). (2) Pin n8n and Postgres to an EU region. (3) Log every AI call's input/output to the n8n execution log for at least 90 days — this is necessary for DSAR responses (data-subject access requests) which Aryze, as infrastructure, will need to be able to support.

Recording consent. Linked to the opt-in design in §5.3 but worth calling out separately: in several EU jurisdictions (Germany, France, Denmark to a lesser extent) recording a meeting without all parties' explicit consent is unlawful, not just impolite. The opt-in mechanism handles the technical side; the conversational side is Rabii's responsibility — open the meeting with "I'm recording this for note-taking, any objections?" The visible-bot tools (Fireflies, Gong) make consent natural because the bot is on the attendee list; Granola's invisibility means Rabii must remember to state it out loud. Either way: never record without consent, and never process a recording where consent was refused or unclear.

Brand and capability discipline. This is an Aryze-specific risk: AI-drafted CRM notes or emails that promise things Aryze can't deliver, or that use forbidden phrases. Mitigations: the synthesis prompt loads the Product-Description files as ground truth and the forbidden-phrases list as a regex screen. Anything flagged is held for Rabii review. The 13 May audit report (Documents/Reports/Audit-Modsigelser-2026-05-13.md) is the reason this is non-negotiable.

Vendor concentration. Microsoft Graph + Claude API + one CRM + one transcript tool + n8n is five vendors. If Claude API goes down, classification stops; if Graph goes down, inbox stops. Mitigation: every workflow is designed to degrade gracefully (n8n queues retries, no critical path requires synchronous AI). Outages are detectable in the n8n monitor; we are not building a system that fails silently.

Cost. Claude API costs scale with email volume and transcript length. Order-of-magnitude estimate (to be verified): Sonnet at EUR 3 per million input tokens + EUR 15 per million output tokens; 200 emails/day × ~500 tokens average input = 100,000 tokens/day = EUR 0.30/day on classification. Transcripts at 5,000–15,000 tokens each, 4–6 per week, on Opus = ~EUR 5–10/week. n8n VM ~EUR 10/month. Total marginal cost: under EUR 100/month at current volumes. CRM API costs depend on the CRM's pricing.

Audit trail. Every AI call has its prompt, input, output, and decision logged in the n8n execution log. Every CRM write has an Activity record. Every classification has the Outlook Category as the artefact. The system is fully traceable — important both for debugging and for Compliance.

Human-in-the-loop sunset. The 48-hour review window for the first month is deliberate. After month one, review at ≥80 confidence is automated and only <80 is queued. Rabii can extend or shorten this; it's a configuration not a structural choice.

8. Open questions before we build

  1. Which CRM? And does its API support webhooks (push) or only polling? Webhook capability decides whether dashboard updates are real-time or 5-minute-delayed.
  2. Which transcript tool? Fireflies, Gong, Granola, and Teams Premium are all possible. The decision now has two axes: (a) integration effort, and (b) opt-in posture — see §5.3. Granola and Teams Premium are the cleanest from an opt-in standpoint; Fireflies has a visible bot; Gong is too heavy for Aryze's stage. If Teams Premium, transcripts are inside M365 and we save one connector.

    2a. What opt-in signal do you want to use? Three options, not mutually exclusive: (i) explicit per-meeting action by Rabii (click record in Granola / invite Fireflies bot / click record in Teams), (ii) a title tag like [Sales] or [Aryze] that the orchestrator looks for, (iii) a dedicated "Aryze Sales" calendar that the bot watches. Recommendation: (i) as the primary, with (iii) as backup for cases you keep on a separate sales calendar already.
  3. Are there existing CRM custom fields we should map to, or are we adding the entire field set from §4? Field additions need CRM admin time.
  4. Self-host n8n or use Make.com? Self-host is cheaper and more auditable; Make is faster to start. Recommendation: self-host, but if there's no appetite to run a small VM, Make.com is acceptable for Phases 1–4.
  5. Who manages the Azure AD app registration? This needs an admin on the Aryze M365 tenant. Identify the right person early — this is often the slowest piece in Phase 0.
  6. DPA with Anthropic signed? If not, raise with legal before Phase 2 (transcript content includes customer PII).
  7. Inbox volume. Roughly how many emails/day does Rabii receive? Determines whether Sonnet costs are EUR 0.30/day or EUR 3/day. Doesn't change architecture.
  8. Outbound emails (sent by Rabii) — should they also create Activity records? If yes, we need Graph Mail.Send-shaped subscription on the Sent Items folder (read, not send). Recommended yes; it gives a complete picture.
  9. Multi-user from the start, or just Rabii? Architecture is the same; permission scoping changes if other Aryze sellers come on later.
  10. Naming. What should we call this internally? Suggest: "Aryze Sales OS" (operational; not "AI platform" which would conflict with brand voice). Open to alternatives.

9. What we're explicitly not building (and why)

  • Auto-send of customer emails. Brand and capability discipline make this too risky for a small team. AI drafts; humans send.
  • Auto-stage transitions. A stage change is a meaningful event. Letting AI move a deal from discovery to scoping based on email tone strips that meaning. The AI proposes; Rabii confirms.
  • Lead scoring beyond P1–P4. A 0–100 score is precision theatre. Four bands map to "today / this week / this month / background" — that's how humans actually plan.
  • Native dashboard build. The Cowork artifact reading the CRM gets us 80% of the value for 10% of the build effort. If a real BI tool becomes necessary in 12 months, we can move to it then.
  • Account-level intelligence pulls (Common Room, Apollo signals). Plugins exist but aren't authenticated yet, and the value is unclear until basic workflows are in. Revisit at Phase 8.
